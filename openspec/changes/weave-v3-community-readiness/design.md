# Design — weave-v3-community-readiness

## Architecture Overview

v3 introduces a planning/process architecture centered on deterministic evidence and governance gates.

This v3 delta also adds internet-catalog planning with a strong consistency contract across initial sources.

### Core design tracks

1. **Evidence pipeline (Option A)**
   - Required CI workflows emit normalized per-run JSON evidence artifacts.
   - A `workflow_run` aggregator reads those artifacts and consolidates them by head SHA.
   - Aggregator emits deterministic manifest payload (`openspec/evidence/<sha>.json` as artifact reference) and publishes PR-facing summary.

2. **Debt-first closure lane (Batch 0)**
   - `DEF-V2-001`: repository hygiene enforcement around Python cache artifacts.
   - `DEF-V2-002`: migration from manual evidence capture to manifest-driven evidence references.
   - `DEF-V2-003`: PR checklist/label/issue-link coherence gate.

3. **Version governance lane**
    - Semver policy map and bump decision matrix.
    - Release validation gate ensuring `internal/cli/version.go` and version tests move together.

4. **Catalog internet consistency lane**
   - Multi-source provider ingestion (`github_curated_index`, `registry_json`).
   - Deterministic normalization schema and stable identity/dedup pipeline.
   - Snapshot-bound repeatable search semantics (online/offline parity on same snapshot).
   - Trust policy enforcement (allowlist + pin-by-ref/hash) and malformed-data quarantine.

## Deterministic Evidence Manifest Model

### Inputs
- Per-workflow JSON artifacts generated by required checks.

### Consolidation key
- `head_sha` (single manifest per commit SHA).

### Determinism rules
- Stable schema version.
- Stable key ordering in serialized JSON.
- Stable ordering of workflow entries (for example lexicographic by workflow ID/name).
- Explicit missing/failed workflow records rather than omission.

### Output channels
- Artifact reference keyed by SHA (for example `openspec/evidence/<sha>.json`).
- PR comment summary that links the same deterministic manifest reference.

## PR Quality Gate Coherence Model

The gate evaluates three dimensions:

1. Checklist state completeness against policy-required items.
2. Label coherence relative to checklist declarations.
3. Issue-link semantics (`Closes #<id>` or explicit `N/A`).

Failure mode is hard-gate (non-zero), with actionable mismatch diagnostics.

## Version Governance Model

### Policy contract
- Patch: backward-compatible fixes/docs/chore with no API/behavior expansion.
- Minor: backward-compatible feature additions.
- Major: breaking behavior or schema contract changes.

### Enforcement
- Release/validation pipeline verifies:
  - bump rationale presence,
  - `internal/cli/version.go` updated,
  - version tests updated/passing,
  - release metadata consistency with declared semver bump.

## Catalog Internet Consistency Model

### Source provider abstraction

- `github_curated_index`: resolves curated catalog index pinned by immutable ref/hash.
- `registry_json`: resolves structured registry JSON pinned by immutable revision/hash.

Both providers emit raw records into a shared normalization pipeline.

### Deterministic normalization schema

All records are normalized into canonical entry shape with required fields:

- `entity_type` (`skill` | `command`)
- `canonical_name`
- `provider_source`
- `provider_ref`
- `version_ref`
- `description`
- `tags` (canonicalized deterministic ordering)
- `updated_at`

Normalization determinism rules:

- Required field validation before merge.
- Whitespace, casing, and alias canonicalization rules are fixed by schema version.
- Tags and multivalue fields are sorted deterministically.

### Stable identity and dedup

Stable identity key:

- `identity_key = hash(entity_type + canonical_name + normalized_primary_locator)`

Dedup/merge policy for identical identity keys:

1. Drop records from non-allowlisted or unpinned sources.
2. Prefer fully valid normalized record over partial record.
3. Prefer freshest trusted `updated_at` within allowed staleness window.
4. Tie-break by configured provider precedence.
5. Final tie-break by lexical (`provider_source`, `provider_ref`).

No valid winner => `quarantined` state with structured diagnostics.

### Deterministic ranking/sorting

Result ordering contract for a query on a given snapshot:

1. Primary: ranking score from deterministic scoring function (pure over query + canonical fields + policy config).
2. Secondary: lexical `canonical_name`.
3. Tertiary: lexical `identity_key`.

No source-time randomness or non-deterministic iteration may affect order.

### Snapshot-based sync and offline parity

Sync transaction output:

- `snapshot_version` (schema and policy version)
- `snapshot_hash` (content-addressed digest of canonical entry set + policy metadata)
- `generated_at`

Search behavior:

- Query execution is bound to one committed snapshot.
- Online and offline search over same `snapshot_hash` must produce identical listing set and order.

### Freshness/staleness and conflicts

Each canonical entry carries:

- `freshness_observed_at`
- `source_updated_at`
- `staleness_status` (`fresh` | `stale` | `unknown`)
- `conflict_state` (`none` | `resolved` | `quarantined`)

Conflict resolution uses deterministic policy above; unresolved conflicts become quarantined with diagnostics.

### Malformed/partial data behavior

- Malformed source records: skip + emit diagnostics.
- Partial records missing required canonical fields: quarantine candidate and continue snapshot build.
- Snapshot commit is atomic: only validated canonical set can become active snapshot.

## Risks and Mitigations

- **Risk**: Aggregator misses late workflow completion for same SHA.  
  **Mitigation**: rerunnable consolidation step keyed by SHA with idempotent overwrite semantics.

- **Risk**: PR template changes break checklist parsing.  
  **Mitigation**: parser contract tests tied to canonical template snapshots.

- **Risk**: Version policy interpreted inconsistently by contributors.  
  **Mitigation**: explicit decision matrix in PRD + CI failure messages with examples.

- **Risk**: Equivalent entries from multiple sources diverge in subtle fields and cause listing drift.  
  **Mitigation**: fixed canonical schema + stable identity key + deterministic merge tie-break chain.

- **Risk**: Source freshness drift causes inconsistent outputs between online and offline.  
  **Mitigation**: strict snapshot-bound query semantics and explicit freshness metadata in responses.

- **Risk**: Supply-chain risk from mutable/untrusted internet sources.  
  **Mitigation**: allowlist and pin-by-ref/hash enforcement before normalization.
